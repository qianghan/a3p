📄 Feature Plan · v1.0 · 2026-06-29

Past Tax Filing Upload

Upload historical tax return PDFs (T1, T4, NOA, 1040, W-2, 1099), extract structured data via Gemini Vision, store originals in Vercel Blob, surface them in the Tax Plugin UI and chat, and use them to pre-fill and advise on the current year's return.

Jurisdictions: Canada + US
Extraction: Key totals + per-form detail
Plugin: agentbook-tax (port 4053)
Storage: Vercel Blob
Parser: Gemini Vision (PDF → images)

What's New vs What Already Exists

The tax plugin already has a sophisticated slip (T4/T5/RRSP) OCR pipeline, a NETFILE-ready e-filing skeleton, and a year-end tax package generator. This feature builds on top of that foundation — it doesn't replace any of it.

Data Model

One new Prisma model, one new Prisma enum (as a string field), and a minor extension to AbTaxFiling.

New: AbPastTaxFiling

model AbPastTaxFiling {
  id            String   @id @default(uuid())
  tenantId      String

  // What it is
  taxYear       Int              // 2023, 2024, etc.
  jurisdiction  String           // 'ca' | 'us'
  region        String?          // 'ON' | 'CA' | 'NY' (province / state)
  formType      String           // 'T1' | 'T4' | 'T4A' | 'T5' | 'NOA' | 'T2125'
                                 // | '1040' | 'W-2' | '1099-NEC' | 'K-1'

  // Storage
  blobUrl       String           // Vercel Blob permanent URL (PDF)
  blobKey       String           // pathname for deletion

  // Extraction
  extractedData Json  @default("{}")  // see Extracted Fields section below
  confidence    Float @default(0)     // 0–1, LLM self-reported
  status        String @default("uploaded")
                                 // uploaded | parsing | confirmed | error

  errorMsg      String?          // if status = 'error'
  notes         String?          // user-added annotation

  createdAt     DateTime @default(now())
  updatedAt     DateTime @updatedAt

  @@index([tenantId, taxYear])
  @@index([tenantId, jurisdiction, formType])
  @@schema("plugin_agentbook_tax")
}

Step 5 — PDF rendering strategy

Gemini Vision accepts images, not raw PDFs. We need to convert PDF pages to images before sending. Two options:

Option A — pdf-to-img (Node)

Use pdf2pic (wraps graphicsmagick or pdftocairo) to render pages to PNG server-side. Works locally and in most Node environments.

⚠️ Requires native binaries — not available in Vercel Functions by default. Works in the standalone backend (port 4053).

Option B — send PDF URL directly to Gemini

Gemini gemini-1.5-pro and later models accept PDF files natively via the File API or inline base64. Pass the Blob URL directly.

✓ No native deps — cleanest path. Gemini handles multi-page extraction.

Recommendation: Option B. Pass the Vercel Blob URL (or base64 of the PDF) directly to Gemini 1.5 Pro via the inlineData part with MIME type application/pdf. Gemini reads all pages and extracts structured data in a single call. This avoids binary dependencies and works in both local backend and Vercel serverless.

File constraints

  • Accepted MIME types: application/pdf only
  • Max size: 20 MB per file
  • Blob key pattern: tax-filings/{tenantId}/{taxYear}/{formType}-{timestamp}.pdf
  • Blob access: private — signed URLs or proxy through API

PDF Parsing — Gemini Prompts

The jurisdiction-aware parsing agent uses a TaxDocumentParser class in tax-past-filings.ts. It selects the right system prompt based on detected or user-specified form type.

Step 1 — Form Identification

SYSTEM: You are a tax document classifier. Given the first page of a tax document,
identify:
1. The form type: T1 | T4 | T4A | T5 | T3 | NOA | T2125 | RRSP | 1040 | W-2 | 1099-NEC | K-1 | other
2. The tax year (4-digit year on the form)
3. The jurisdiction: 'ca' (Canadian CRA form) or 'us' (IRS form)
4. The province/state if visible

Respond as JSON only:
{ "formType": "T1", "taxYear": 2024, "jurisdiction": "ca", "region": "ON" }

Step 2 — Extraction (jurisdiction-specific)

Canada T1 General

SYSTEM: You are a Canadian CRA T1 General tax return parser.
Extract the following line numbers from this T1 return (all monetary values in CENTS — multiply dollars by 100):

Key lines to find:
- Line 10100: Employment income
- Line 13500: Business income (net)
- Line 15000: Total income
- Line 23200: Net income before adjustments
- Line 26000: Taxable income
- Line 43500: Basic federal tax
- Line 48200: Total federal/provincial tax payable
- Line 48400: Refund amount (blank if balance owing)
- Line 48500: Balance owing (blank if refund)
- RRSP deduction limit for following year (usually on bottom of page 4 or Schedule 7)

Also extract any attached T2125 (business income/expenses) totals if present.

Return JSON matching this schema exactly:

  "jurisdiction": "ca",

All null values are acceptable if a line is blank or not present on the form.

  • "Do I still have RRSP room from 2023?" → pulled from NOA or T1 extraction
  • "Will I owe more or less than last year?" → compare current YTD estimate vs prior year tax

Current Year Pre-fill

The GET /past-filings/prefill?year=2025 endpoint returns a pre-fill map derived from the most recent confirmed past filing (year 2024 for a 2025 return). It maps extracted fields → current T1/1040 field IDs used in AbTaxFormTemplate.

Canada T1 pre-fill map

Past Filing Field→Current T1 Field (AbTaxFormTemplate)Notes
T2125.homeOfficePct→home_office_pctLikely same % YoY
T1.region→Province (TaxConfig)Auto-set if not already set
T2125.businessName→business_nameDefaults only if blank
T2125.industryCode→industry_codeSuggests, user can change
NOA.rrspDeductionLimit→Shown in advisor RRSP contextNot a T1 field but surfaced in UI
Pre-fill never overwrites existing filled fields. It only populates fields that are currently empty (missing) in the current year's AbTaxFiling.forms. The user must confirm pre-filled data.

Pre-fill flow

  1. User clicks "Pre-fill 2025 return from 2024 T1" in the UI
  2. Frontend calls GET /past-filings/prefill?year=2025
  3. Backend returns field map → UI previews the suggestions
  4. User reviews + confirms → PATCH /filing/2025/fields called for each accepted field
  5. AbTaxFiling.prefillSourceYear updated to record provenance

E-Filing — Canada

CRA's NETFILE allows direct electronic filing. Several integration levels exist — ordered from simplest/free to most complex/powerful.

1

NETFILE XML Export (Free — Recommended MVP)

Generate a CRA-spec NETFILE XML file from the user's completed T1 data in AbTaxFiling. User downloads the XML and submits it themselves at canada.ca/netfile, or imports it into any NETFILE-certified app (StudioTax, CloudTax, UFile — all free for simple returns).

Implementation: Add /tax/export-netfile-xml endpoint that serializes AbTaxFiling.forms into the CRA XML envelope format (schema published by CRA annually). No CRA certification needed to generate the XML — only to submit it directly. Estimated effort: 2–3 days.

Free · No CRA agreement · Works today
2

EFILE for Tax Preparers via CRA Represent a Client API (Free for Authorized Reps)

AgentBook acts as an authorized representative. The user authenticates via "Represent a Client" (RA) on CRA My Account, grants AgentBook access. AgentBook then submits via EFILE using the T1013 authorization. CRA offers this to authorized accounting software — no NETFILE certification needed, but requires T1013 form flow.

Effort: Medium (CRA OAuth + T1013 flow). User friction: Requires CRA My Account login. Best for premium tier.

Free but moderate effort · Premium tier
3

NETFILE Software Certification (Direct Submission)

Apply to CRA's NETFILE Certified Software Program. Upon certification, AgentBook can call the NETFILE web service directly to submit T1 returns on behalf of users. Requires CRA review (6–12 months), security audit, and annual recertification. No dollar fee from CRA, but significant developer time.

Best for: Premium tier at scale (>1,000 filings/year).

Free CRA fee · High effort · Months of review
4

Partner API — TaxCycle or Profile/Cantax (Paid)

Enterprise tax software APIs that are already NETFILE-certified. Possible white-label integration for premium B2B users (accounting firms using AgentBook). Per-return pricing.

Paid per return · B2B only

E-Filing — United States

1

IRS MeF XML Export (Free — Recommended MVP)

Generate an IRS Modernized e-File (MeF) compliant XML from the user's 1040 data. User gives this XML to their CPA who submits via ERO (Electronic Return Originator) software. Or user manually uploads to IRS Free File Fillable Forms. IRS publishes the MeF schema annually — no special certification needed to generate the XML.

Effort: 2–3 days for 1040 schema, more for schedules (C, SE, etc.).

Free · No IRS agreement needed · Works today
2

IRS Free File Alliance (Conditional Free)

IRS Free File partners offer free e-filing for taxpayers with AGI ≤ $79,000 (2025). Becoming a Free File Alliance member requires an agreement with IRS + significant compliance overhead. Not practical as a standalone integration for AgentBook.

Alternative: Deep-link users to a partner (FreeTaxUSA, OLT) with pre-filled data via URL params — some partners accept this.

Complex · IRS agreement needed
3

Direct IRS EFIN / MeF Submission (Premium)

Obtain an EFIN (Electronic Filing Identification Number) from IRS, become an ERO or ESSP. Allows direct MeF submission. Heavy compliance burden — requires annual IRS suitability check, security plan, etc. Realistic only at significant scale.

High compliance burden · Scale required
4

FreeTaxUSA / TaxSlayer White-Label API (Paid Partner)

Several mid-tier tax software vendors offer white-label or data-passthrough APIs. Revenue share or per-return pricing. Best for a premium AgentBook tier where users want 1-click filing without CPA involvement.

Revenue share · Partner agreement needed

Jurisdiction Extensibility — Add NZ, UK, AU Without Core Changes

The upload, blob storage, DB, and parsing pipeline are jurisdiction-agnostic. All jurisdiction-specific knowledge lives in a PastFilingPack interface added to packages/agentbook-jurisdictions/src/interfaces.ts — the same pattern already used by TaxBracketProvider, SalesTaxEngine, etc. Adding a new country = implement the interface in a new directory. Zero changes to core pipeline code.

New interface: PastFilingPack

  parseExtraction(raw: any, formType: string, taxYear: number): StandardTaxExtract

  // Map extracted data → current-year form fields for pre-fill
  preFillMap(extract: StandardTaxExtract): PreFillSuggestion[]

  // Plain-English summary for injection into Gemini advisor prompts
  summarize(extract: StandardTaxExtract): string

  // Optional: generate e-file export (XML, JSON) from completed AbTaxFiling
  generateEFileExport?(forms: Record<string, any>, taxYear: number, region?: string): EFileExport
}

Pack registry (loader)

packages/agentbook-jurisdictions/src/past-filing-loader.ts — a thin registry that maps jurisdiction codes to pack instances: 

// packages/agentbook-jurisdictions/src/past-filing-loader.ts

import { CaPastFilingPack } from './ca/past-filing-pack.js';
import { UsPastFilingPack } from './us/past-filing-pack.js';
// New countries: import here and add one line below. Nothing else changes.

const PACKS: Record<string, PastFilingPack> = {
  ca: new CaPastFilingPack(),
  us: new UsPastFilingPack(),
};

export function getPastFilingPack(jurisdiction: string): PastFilingPack {
  const pack = PACKS[jurisdiction];
  if (!pack) throw new Error(`No PastFilingPack for jurisdiction: ${jurisdiction}`);
  return pack;
}

export function listSupportedJurisdictions(): string[] {
  return Object.keys(PACKS);
}

Resolved Design Decisions

#DecisionResolved
1 Blob access: private or public? Private blobs — tax documents are sensitive. Download proxied through GET /past-filings/:id/download which generates a short-lived Vercel Blob signed URL and redirects. Blob URLs never exposed directly to the frontend.
2 Parsing: sync or async? Async — upload returns immediately with status uploaded. Parse runs in the background (same request context using waitUntil on Vercel, or a deferred call on the standalone backend). UI polls GET /past-filings/:id every 3s, shows spinner until confirmed or error.
3 Upload quota per tenant? No hard limit initially — Vercel Blob is cheap at this scale. A checkQuota() hook is stubbed in the upload handler (same pattern as @naap/billing used elsewhere), so a billing tier limit can be wired in later without touching core upload logic.
4 Multi-form PDFs (T1 + T4 bundled)? Supported — Gemini reads all pages and extracts whatever it finds. The single AbPastTaxFiling record stores the full PDF + extraction. User can upload again separately for cleaner per-form records, but bundled uploads are not rejected.
5 Where does e-filing XML generation live? Tax backend (port 4053) — owns all filing data and the PastFilingPack.generateEFileExport() interface. Exposed via GET /tax/export/netfile-xml and GET /tax/export/mef-xml, proxied through Next.js at /api/v1/agentbook-tax/.
6 Pre-fill: automatic or manual? Manual — user explicitly triggers pre-fill via "Pre-fill 2025 return from 2024 T1" button. A modal shows each suggestion with accept/reject per field before any change is applied. Automatic would feel invasive for tax documents.
