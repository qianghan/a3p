# Personal Finance — Net-Worth Trends + Proactive Nudges (PR-2 of the personal-finance/tax-filing launch program)

## Context

PR-1 (merged, `#240`) shipped the transactions/budgets UI and a chat write-skill for the personal-finance plugin, closing the "you can't record anything without the API" gap. This is PR-2: the first genuinely premium personal-finance feature, gated behind a new `hasAddOn()` check.

*Revision note (post-adversarial-review, before this spec went to a plan): the first draft claimed this would be "the first route in the repo to enforce `hasAddOn()`." That's wrong — `hasAddOn()` is already enforced in at least six `agentbook-startup` routes, `lib/agentbook-student/guard.ts`'s `requireStudentAddon()` helper, and the chat pipeline (`server.ts:3220`). The corrected plan is to reuse that existing guard-helper pattern (a new `requirePersonalInsightsAddon()` mirroring `requireStudentAddon()`), not to invent enforcement from scratch. The draft also claimed nudges would reach Telegram "matching `tax_deadline`'s existing channels" — also wrong: `createNotification()`/the `AbEvent` `calendar-check` writes never reach Telegram; `tax_deadline` content only reaches Telegram because a separate cron, `morning-digest`, independently recomputes what's due and calls `sendToAllChannels()`. Both corrections are reflected below.*

Decisions already made with the user before writing this spec:
1. Nudges are delivered via the same two-channel pattern several existing proactive skills already use in production (`proactive-alerts`, `home-office-quarterly`): `createNotification()` for the dashboard bell/inbox, **and** a direct `sendToAllChannels()` call for Telegram — not the fully-orphaned `packages/agentbook-framework` proactive-handlers engine (22 handler files, zero live callers, never run in production), and not `calendar-check`'s `AbEvent`-only pattern (which, on inspection, has no Telegram consumer at all).
2. Billing gate is a new dedicated add-on (`personal_insights`), following the `student_success` pattern exactly, independent of the tenant's base plan tier.
3. v1 ships all three nudge triggers: budget-threshold crossed, net-worth monthly change, negative savings rate.
4. Trend window is trailing 12 months, monthly points.
5. Nudges deliver to both dashboard and Telegram (via the corrected mechanism in decision 1, not `calendar-check`'s).

## Goal

Give a subscribed tenant a 12-month net-worth trend chart and three proactive nudge types, gated behind a new `personal_insights` add-on — while keeping the existing free `personal-snapshot` current-state chat answers (net worth right now, this month's spending) unaffected.

## Scope

**In scope:**
1. **Trend computation** — a pure function that reconstructs net worth at each of the trailing 12 month-ends from existing data. Base formula: `account.balanceCents − Σ(that account's transactions dated after the month-end)`, applying the same asset/liability sign handling `computeSnapshot()` already does at "now" (`lib/personal-snapshot.ts`) at *each* month-end, not just the current point in time. Three correctness rules, verified against the real `AbPersonalAccount`/`AbPersonalTransaction` routes during review, apply on top of the base formula:
   - **Clamp to account age.** For any month-end before `account.createdAt`, that account contributes `$0`, not its full starting balance — otherwise an account's opening balance appears to have existed in months before the account did.
   - **Archived accounts are excluded from the reconstruction entirely**, including from months while they were still active. `AbPersonalAccount` has no `archivedAt` timestamp, only a boolean `archived` flag, so there's no way to know when an account was archived without a schema change — out of scope for this PR. This is a documented, accepted limitation (a tenant who archives an account will see a discontinuity in their trend chart), not a bug to chase down here.
   - **Manual balance corrections (the accounts `PUT` route sets `balanceCents` directly, with no compensating transaction) are a second documented, accepted limitation** for the same reason — reconstructing "what the balance would have been" through an untracked manual edit isn't solvable without either forbidding direct balance edits (a bigger, unrelated change to PR-1's shipped route) or recording an adjusting transaction on every edit (a scope increase not justified by this feature). The task should double-check whether the Plaid-sync cron (`plaidAccountId` field, referenced by a sync job) also mutates `balanceCents` directly outside a transaction; if so, the same accepted-limitation note applies there too.
   No new schema for the historical data itself; this works immediately for every tenant with any transaction history, subject to the two documented limitations above.
2. **New model** `AbPersonalNudgeLog` — dedup guard so the same nudge doesn't re-fire every time the check runs within the same period. Fields: `id, tenantId, nudgeType, periodKey (e.g. "2026-07"), category (nullable, budget nudges only), createdAt`, unique on `(tenantId, nudgeType, periodKey, category)`.
3. **New billing add-on** `personal_insights` — new `BillAddOn`/`BillAddOnPrice` rows via a new seed script `bin/seed-personal-insights-addon.ts` (mirrors `bin/seed-student-success-addon.ts`: `isActive:false` by default, `ACTIVATE=1` to flip, single `'standard'` tier, `us`/`ca` regional pricing — same $49 USD / $65 CAD precedent as `student_success` unless told otherwise).
4. **A shared gate helper**, `requirePersonalInsightsAddon(request)` in a new small module (mirroring `lib/agentbook-student/guard.ts`'s `requireStudentAddon()` exactly — same `{tenantId} | {response: 402-with-upsell-body}` return shape), used by every enforcement point below instead of each one inlining its own `hasAddOn()` call. This is the established pattern for this kind of gate in this codebase, not a novel mechanism.
5. **New API route** `GET /api/v1/agentbook-personal/trend` — returns the 12 monthly net-worth points, using `requirePersonalInsightsAddon()`.
6. **Nudge checks, folded into the existing daily `morning-digest` cron** (`apps/web-next/src/app/api/v1/agentbook/cron/morning-digest/route.ts`) rather than a new hourly cron route. This fixes two problems found on review: (a) `morning-digest` already loops every tenant once a day and already calls `sendToAllChannels()`, so extending it delivers Telegram correctly for free and avoids a new cron registration; (b) an hourly cadence was never justified for these checks — a month-over-month net-worth comparison and a monthly savings-rate check don't need 24x-a-day resolution, and even the budget-threshold check (the one nudge type where faster reaction has some value) doesn't need hourly precision for a v1. For every tenant with an active `personal_insights` subscription, `morning-digest` additionally: checks each `AbPersonalBudget` for a newly-crossed 80%/100% threshold, checks this month's net worth vs. last month's (two points from the trend computation — not a full 12-month recomputation), checks this month's income vs. spending for a negative savings rate — each gated by an `AbPersonalNudgeLog` dedup check, and each firing **both** `createNotification()` (dashboard) and `sendToAllChannels()` (Telegram) when it fires.
7. **New notification categories** — `budget_alert`, `net_worth_update`, `savings_warning` added to the existing `NOTIFICATION_CATEGORIES` union (`apps/web-next/src/lib/notifications.ts`), NOT added to `COMPLIANCE_LOCKED_CATEGORIES` (these are value-add nudges, not mandatory like `tax_deadline` — a tenant can opt out via the existing notification preferences tab). Confirmed on review: the preferences **API** (`.../notifications/preferences/route.ts`) derives its list from the shared const, so the 3 new categories become functional opt-out toggles automatically. Two `.tsx` files hardcode display *labels* only and need the 3 new entries added for polish (both have safe fallbacks to the raw category slug, so this is cosmetic, not a correctness requirement): `components/settings/AgentBookSettingsPanel.tsx`'s `NOTIFICATION_CATEGORY_LABELS`, and `app/(dashboard)/notifications/page.tsx`'s `CATEGORY_LABEL`.
8. **Extend the existing `personal-snapshot` chat skill** (not a new skill) to recognize trend-shaped phrasing and answer from the new trend endpoint via `requirePersonalInsightsAddon()` (a clear, on-brand upsell message if not subscribed, matching the tone of the existing scholarship/career/housing "part of X — enable it in Settings" precedent, not a bare 402 message). Routing and free/paid classification, pinned down precisely (see Design decisions below) to avoid the exact class of routing bug found in PR-1's reviews. MCP parity for this is automatic — the MCP surface is a single generic tool that proxies free text to the same skill router with `channel:'mcp'`, so no separate MCP work is needed, but the test plan includes an explicit `channel:'mcp'` case rather than assuming this silently.
9. **UI**: a trend chart section on `/personal`, gated with a teaser/upsell card for non-subscribers — following the same `student_success`-gated-plugin visibility precedent (a visible-but-locked card, not simply hidden) rather than invisibility. A minimal purchase entry point (link to the existing add-on purchase route already used by `student_success`, `apps/web-next/src/app/api/v1/agentbook-billing/me/addons/[code]/subscribe/route.ts` — reused as-is, no new billing UI needed beyond wiring the `personal_insights` code through it).

**Out of scope (explicitly deferred):**
- Household/spousal sharing of trend data (PR-6+ territory per the gap review, not part of this program's launch scope).
- Any change to the free `personal-snapshot` current-state answers.
- Multi-jurisdiction currency work beyond what PR-1 already established (`jurisdiction-currency.ts` is reused as-is for the chart's axis/tooltip formatting).
- Retroactively enforcing `hasAddOn()` anywhere else in the repo (e.g. `tax_package_generation`) — this PR only wires the gate for its own new surface; other unenforced flags are a separate, already-flagged item (PR-5 touches `tax_package_generation` specifically).
- Fixing the two documented reconstruction limitations (archived-account timing, manual balance-edit drift) — accepted for v1, noted above.
- A one-time "you crossed your budget — `personal_insights` would have warned you" upsell nudge to non-subscribers. Plausibly a strong conversion lever (flagged during review), but it's a product/growth decision outside this spec's scope, not an engineering default to bake in unasked.

## Design decisions

- **Reconstruction, not a snapshot table.** A monthly-snapshot cron would only start accumulating history from the day it first runs — meaning existing tenants like Maya would see a flat/empty chart for months. Reconstructing historical net worth from existing transaction rows means the chart is immediately populated and correct for any tenant with transaction history, at the cost of a somewhat more involved query (sum transactions after each month-end boundary, per account, per month) rather than a cheap snapshot read. Given `AbPersonalTransaction` volumes are small per tenant (personal finance, not a high-volume ledger), this is not a performance concern.
- **One add-on, three nudge types, one endpoint — not three separate gates.** Keeps the billing surface simple: a tenant either has `personal_insights` or doesn't; there's no need for granular per-nudge-type purchases in v1.
- **`AbPersonalNudgeLog` is intentionally minimal** — no soft-delete, no update path, just an insert-only dedup log. A nudge either fired for a given tenant/type/period/category or it didn't; there's nothing to edit.
- **Extending `personal-snapshot` over a new skill** keeps the free/paid split legible in one place (one skill, one gate check inside it) rather than splitting personal-finance chat behavior across two skills that could drift or double-trigger, mirroring the "don't add a skill when extending an existing one is cleaner" judgment already applied elsewhere this session.
- **Trend-trigger phrasing must always be net-worth/personal-anchored, never a bare temporal phrase.** Review caught that a phrase like "over the last year" alone, with no anchor, risks colliding with unrelated skills depending on DB row order (the same first-match-wins/no-`orderBy` hazard PR-1's reviews found repeatedly) — e.g. a business-side "how has revenue trended over the last year" question. Every new trigger must combine a personal/net-worth cue (reuse `personal-snapshot`'s existing anchors: `net worth`, `personal`, `household`, `savings rate`) together with a comparison/temporal cue (`trended`, `over time`, `compared to`, `vs last month`, `change`) — never the temporal cue alone. Mirror `excludePatterns` the same way `record-personal-transaction`'s collisions were resolved in PR-1 (shared pattern constants, not ad hoc per-skill regexes).
- **Inside the handler, default to the free/current-state path; only clear temporal language upgrades to the gated trend path.** This is the safe failure direction: if the sub-classifier misfires, the worst case is a subscribed user has to rephrase to get trend data (a UX papercut) — never a non-subscriber silently getting the paid answer, and never an accidental gate on an ordinary free query (which would be a real regression of PR-1's shipped behavior). The classifier itself: if the matched trigger phrase also contains one of the comparison/temporal cues above, treat as a trend query (gated); otherwise treat as current-state (free, unchanged from PR-1). Test both misfire directions explicitly, not just the two happy paths.
- **Threshold values** are implementation-owned, not re-litigated with the user here, but pinned down concretely so independent implementation tasks don't diverge: budget nudges fire at 80% and 100% of the category limit (two distinct dedup periods, both keyed off the same `periodKey`+`category`, so crossing 80% then later 100% in the same month fires twice, not zero or once-only); the net-worth nudge fires when the month-over-month change exceeds `max($100, 5% of prior month's net worth)` in either direction (a $50 swing on a $500 net worth is noise, the same $50 swing on a $50,000 net worth is not — a pure percentage or pure absolute floor alone gets one of those two cases wrong); the savings-rate nudge fires whenever this month's income minus spending is negative, no threshold banding. All three are easy to tune later without a design change — the exact numbers are a config concern, not an architectural one.

## Test plan

- Unit: month-end net-worth reconstruction math (multiple accounts, transactions before/after boundaries, an account created mid-window — asserting the `createdAt` clamp specifically, not just "some value"), the nudge checks' threshold/dedup logic (crosses threshold once → fires once; stays crossed → doesn't re-fire same period; drops below and re-crosses next month → fires again), `requirePersonalInsightsAddon()` on the trend route and the extended `personal-snapshot` skill, and the trend-query vs. current-state sub-classifier — both misfire directions explicitly: a current-state phrase must never trigger the gate, and a trend phrase for a non-subscriber must never leak trend data.
- E2E: a subscribed tenant sees the trend chart + gets a real 200 from `/trend`; a non-subscribed tenant sees the teaser card + a 402; a budget-threshold nudge round-trips (create a budget, record a transaction that crosses 80%, run the check directly or via the cron route, assert a notification + `AbPersonalNudgeLog` row exist + `sendToAllChannels` was invoked, assert calling it again doesn't duplicate); the existing free `personal-snapshot` current-state test from PR-1 continues to pass unmodified; an MCP-channel (`channel:'mcp'`) call for both a current-state and a trend query, confirming parity without any MCP-specific code.

## Rollout

Additive schema only (`AbPersonalNudgeLog`, new `BillAddOn`/`BillAddOnPrice` rows) — no destructive migration. Build → unit tests → task-scoped review per task → final whole-branch review → merge to `main` → build + prebuilt deploy → seed the `personal_insights` add-on in production (inactive until explicitly activated, matching `student_success`'s launch precedent) → live e2e verification, matching PR-1's process exactly.
