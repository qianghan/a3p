# Triage Remaining Dependency Vulnerabilities Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Reduce the production-reachable dependency vulnerability count as far as safely possible without a risky blanket `npm audit fix`, continuing directly from PR #241 (already merged), which fixed 3 of 4 critical CVEs and ~40 other vulnerabilities via targeted, named `overrides` — this plan applies the same targeted-override technique to the remaining safe, non-major-bump items.

**Architecture:** Extend the root `package.json`'s existing `overrides` block (already established by PR #241) with pinned versions for every production-reachable vulnerability whose fix is a same-major-line bump — no new tooling, no new pattern, exactly the technique already proven in this repo. A blanket `npm audit fix` was tried and rejected during scoping: it produced an 8,135-line lockfile diff and, confusingly, *increased* the reported count (a known npm-audit quirk from dependency-tree re-flattening) — targeted overrides are the safer, reviewable path, matching PR #241's own approach.

**Tech Stack:** npm workspaces, `package.json` overrides.

## Global Constraints

- No blanket `npm audit fix` / `npm audit fix --force` — every version pin is a named, individually-justified `overrides` entry, mirroring PR #241's precedent exactly.
- Scope to the **42 production-reachable** vulnerabilities (`npm audit --omit=dev`), not the full 59 (dev-tooling-only vulnerabilities — `@nx/*`, jest, eslint-adjacent build tooling not shipped to production — carry much lower real risk and are a lower priority; note this explicitly rather than silently ignoring them).
- Any fix requiring a major version bump of a directly-depended-upon package (`@sentry/nextjs` 9.x→10.26.0) is **out of scope for this PR** — document it as a disclosed, deferred follow-up rather than risking an untested breaking change to error-monitoring configuration. `next` itself is a `isSemVerMajor: false` bump per the audit and IS in scope (same major line).
- After applying overrides, run the full test suite — a version bump that silently breaks something is worse than the vulnerability it fixed.
