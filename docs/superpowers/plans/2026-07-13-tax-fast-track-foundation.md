# Tax Fast-Track Foundation — Implementation Plan (PR-3)

Design: `docs/superpowers/specs/2026-07-13-tax-fast-track-foundation-design.md` (two rounds of adversarial revision — read the "Revision note", every "Revised:" subsection, and the "falsy return, not a thrown exception" paragraph before starting; the fixes there are load-bearing, not cosmetic).

*Plan-level revision note (an independent plan review, before execution): the first draft of this plan resolved several real decisions incorrectly or left them open when they shouldn't have been — a "retry once on version conflict" instruction that contradicts the real precedent (which never retries), a "match parseExtraction's parsing" instruction that was factually wrong about where parsing happens, a mutual-exclusion implementation choice left for the implementer to guess at mid-task despite both options having a non-obvious pitfall, a cancel-keyword list that doesn't actually match the existing regex it claims to mirror, a routing-collision test method mischaracterized as a "node/tsx script" when it's actually an existing, three-times-duplicated vitest suite this PR should consolidate rather than copy a fourth time, and a missing edit to `tax-filing-start` needed to actually prevent the collision the plan's own test checks for. All fixed below.*

## Task 1 — Schema + session data-layer helpers

**Files:**
- `packages/database/prisma/schema.prisma` — add `AbTaxQuestionnaireSession` exactly as specced (`plugin_agentbook_core` schema, `version` field, `consecutiveFailures` field, `@@index([tenantId, status])`).
- `plugins/agentbook-core/backend/src/tax-questionnaire-session.ts` (new) — mirror `agent-planner.ts`'s `createSession()`/`getActiveSession()`/`updateSession()` trio exactly, for the new model:
  - `createTaxQuestionnaireSession(tenantId, taxYear, jurisdiction, region, trigger, sourceFilingId)` — before inserting, expires any other active `AbTaxQuestionnaireSession` for the tenant (mirrors `createSession()`'s own `db.abAgentSession.updateMany({where:{tenantId,status:'active'},data:{status:'expired'}})` shape, applied to this model instead).
  - `getActiveTaxQuestionnaireSession(tenantId)` — `findFirst` where `status:'in_progress', expiresAt: {gt: now}`.
  - `updateTaxQuestionnaireSession(id, version, data)` — raw SQL `UPDATE ... WHERE id=? AND version=?` (`$executeRaw`, self-incrementing `version = version + 1`), returns `false` on mismatch (`result > 0`), exactly matching `agent-planner.ts`'s real `updateSession()` (verified: `agent-planner.ts:254-290`). Callers must re-fetch on `false` — there is no retry built into this helper itself.

- **Resolved now, not left for the implementer to guess mid-task**: the reverse mutual-exclusion direction (starting an `AbAgentSession` must expire any active `AbTaxQuestionnaireSession`) is implemented by **editing `agent-planner.ts`'s real `createSession()` directly** — add `db.abTaxQuestionnaireSession.updateMany({where:{tenantId,status:'in_progress'},data:{status:'abandoned'}})` alongside its existing expire-other-`AbAgentSession` line. This is the one-edit option (both of its two real call sites in `agent-brain.ts`, lines ~915 and ~1007, are covered automatically) versus a call-site edit that would need updating in two places and risks one being missed — exactly the mutual-exclusion hole this whole mechanism exists to prevent. **This edit requires a matching test-infrastructure fix**: `agent-brain-confirm-flow.test.ts`'s `db` mock currently defines `abAgentSession` but not `abTaxQuestionnaireSession` — add `abTaxQuestionnaireSession: { updateMany: vi.fn() }` (or equivalent) to that mock, or the new line in `createSession()` will throw `Cannot read properties of undefined` in every test that exercises it. Do this as part of this task, immediately after adding the `createSession()` edit — don't leave it for Task 3 to discover as a surprise test failure.

**Tests:** unit tests for all three CRUD helpers — creating a session expires a prior in-progress one for the same tenant; `updateTaxQuestionnaireSession` succeeds when `version` matches and fails (returns `false`, no retry) when it doesn't; `getActiveTaxQuestionnaireSession` respects `expiresAt`. Plus: `createSession()`'s existing test suite (`agent-brain-confirm-flow.test.ts`) still passes after the mock update, and a new case confirming `createSession()` now expires an active `AbTaxQuestionnaireSession`.

## Task 3 — `agent-brain.ts` session-recovery branch

**Depends on:** Task 1 (session data-layer helpers, including the now-resolved `createSession()` edit), Task 2 (pack loader).

**Files:** `plugins/agentbook-core/backend/src/agent-brain.ts` — new branch, parallel to (not inside) the existing `AbAgentSession` session-recovery step. Read the spec's full "Revised: exit paths and mutual exclusion" section (and the "falsy return, not a thrown exception" paragraph) before writing this — it is the single most detailed, most load-bearing part of the spec. Specifics, pinned down precisely after plan review:

- **Cancel-keyword detection, checked first.** The real precedent (`agent-brain.ts`'s `CANCEL_RE`) is `/^(cancel|stop|abort|nevermind|n)$/i` — anchored, single-token, case-insensitive. Reuse this exact pattern rather than the spec's illustrative `cancel|stop|never mind` list, extended with one alternation for the two-word phrase form: `/^(cancel|stop|abort|never\s?mind|n)$/i`. On a match: `status:'abandoned'`, no `qaHistory` append, no `expiresAt` refresh, reply confirming cancellation.
- **The success path**: append `{question, answer}` to `qaHistory`, reset `consecutiveFailures` to `0`, refresh `expiresAt` to now+72h, increment `askedCount`, call `pack.nextQuestionPrompt()` → `callGemini(systemPrompt, userMessage, maxTokens?)` (real 3-arg signature, not the spec's illustrative 1-arg example) → fence-strip + `JSON.parse` (Task 2's call-site step) → `pack.parseNextQuestionResponse()`. If the result is `{done:true}` or `askedCount>=8`, mark `completed`.
- **The failure path — check for `callGemini()`'s real non-throwing failure mode.** `callGemini()` returns `string | null` and never throws; a `null`/empty return is itself a failure, checked explicitly (`if (!raw) { ...fail... }`), *separate from* the try/catch around the fence-strip/`JSON.parse`/`parseNextQuestionResponse()` step (which can throw on malformed content). Either failure mode: increment `consecutiveFailures`, no `qaHistory`/`askedCount`/`expiresAt` mutation, reply asking the user to try again, stay `in_progress`; at `consecutiveFailures>=3`, mark `abandoned` instead.
- **Version-conflict handling has no retry — resolved now, contradicting the earlier draft's "retry once" language.** The real `AbAgentSession` branch's own version-conflict handling (`agent-brain.ts`) does not re-fetch or retry — it returns `'Session was modified by another process. Please try again.'` and stops. Mirror that literally: on `updateTaxQuestionnaireSession()` returning `false`, surface the equivalent message for this session type and stop; do not attempt an automatic retry.

**Tests:** the full exit-path matrix from the spec's Test plan section — happy path (question 1 → answer → question 2 → ... → done or cap → completed), cancel keyword (including both `nevermind` and `never mind`) → abandoned with no `qaHistory`/`expiresAt` mutation, expired session treated as inactive, `callGemini()` returning `null` → `consecutiveFailures` increments without other mutations (test this explicitly — not just a thrown exception, since that's the realistic failure mode), a malformed-JSON throw → same failure handling, 3 consecutive failures → abandoned, starting a questionnaire expires an active `AbAgentSession` (via Task 1's `createSession()` edit) and vice versa, a version conflict on update surfaces the no-retry conflict message and does not silently double-process a turn.

## Task 4 — `start-tax-fast-track` skill

**Depends on:** Task 1 (session data-layer helpers) + Task 2 (pack) — **not** Task 3. This skill only ever runs on the *first* message, before any `AbTaxQuestionnaireSession` exists; Task 3's session-recovery branch only runs once one does — the two are on mutually exclusive message paths for a given tenant. All of this task's own tests (routing, confirmed-status filtering, blocked-path confidence, "produces a real first question") exercise `createTaxQuestionnaireSession()` + the pack + a mocked `callGemini`, none of which need Task 3's branch logic to exist yet. **Loosened from an earlier, overly conservative "depends on Task 3" — this task can run in parallel with Task 3, tightening the critical path.** (The genuine multi-turn, real-session-recovery scenario is Task 5's job, which correctly depends on everything.)

**Files:**
- `plugins/agentbook-core/backend/src/built-in-skills.ts` — new INTERNAL manifest, no HTTP endpoint. Triggers per the spec's "Revised: trigger design" section — every trigger must combine filing-intent language with an explicit prior-year/past-filing anchor cue (`last year|past filing|past return|previous filing|previous return`); never a bare "do/file my taxes" phrase alone. Consider using the routing engine's `requirePatterns` (confirmed supported in `skill-routing.ts`) to *require* the anchor cue outright, rather than relying on trigger-phrase construction alone — a cleaner guarantee than pattern-phrasing tricks.
- **Also edit `tax-filing-start`'s manifest** (same file) to add an `excludePatterns` entry for the same anchor cue (`last year|past filing|past return|previous filing|previous return`). This is not optional polish — `tax-filing-start` currently has **no** `excludePatterns` and a broad `'tax.*return'` trigger, so an anchored fast-track phrase like "help me do this year's filing based on last year's tax return" would otherwise match *both* skills simultaneously (routing has no `orderBy`, so which one "wins" would be undefined). Without this edit, the routing-collision test below cannot pass.
- `plugins/agentbook-core/backend/src/server.ts` (or wherever this INTERNAL skill's handler naturally lives, matching the existing convention for other INTERNAL tax skills — check where `tax-filing-start`'s own handler lives and put this alongside it) — filters `listPastFilings()`'s results to `status === 'confirmed'` (do not assume the helper already filters), picks the most recent by `taxYear`; if none, replies pointing at the upload flow at `confidence: 1`; if one exists, calls `createTaxQuestionnaireSession()` (Task 1), builds the first prompt via the jurisdiction pack (Task 2) using the filing's `StandardTaxExtract` and `buildPersonalProfileContext()`'s markdown block, calls `callGemini` (real 3-arg signature), fence-strips + parses, and asks the first question.

**Tests:** the routing-collision verification against the *full* tax-skill family named in the spec (not just 2 skills) — **add these cases to the existing `skill-routing-canonical.test.ts` suite**, not a new script. That file already has the established `shuffle()`/`SHUFFLE_SEEDS`/`pickSkill()` helpers this exact verification technique needs (confirmed present, and already duplicated verbatim across two other test files in this program) — reuse them there rather than reimplementing a third copy. **Worth doing as part of this task**: extract `shuffle`/`SHUFFLE_SEEDS`/`pickSkill` into `plugins/agentbook-core/backend/src/__tests__/helpers/` (a directory that already exists with `mock-llm.ts`/`test-context.ts`) now that this PR would make it a fourth duplicate if left alone — a small, clearly-bounded consolidation, not scope creep. Also: the confirmed-status-filtering test (mixed-status filings, only `confirmed` ever selected); the blocked-path `confidence: 1` test; a happy-path test that a confirmed filing produces a real first question via the mocked pack/LLM; and confirm the new `tax-filing-start` exclude doesn't regress that skill's own existing canonical test cases.

## Process
